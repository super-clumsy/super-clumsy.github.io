git

git是分布式版本控制系统，和集中式版本控制系统有差别。

我个人感觉和比特币有些类似。

但是，这里只谈一些常用的命令

git配置

git的配置文件其实都保存在本地

git config -l 查看git的所有配置文件

git config –system –list 查看git的系统配置文件

git config –global –list 查看git的手工配置文件

git的相关配置文件

  1. Git\mingw64\etc\gitconfig : Git安装目录下的gitconfig –system系统级别
  2. C:\Users\Administrator\ .gityconfig 只适用于当前登录用户的配置 –global全局

这里可以直接编辑配置文件，通过命令行影响后，直接保存在这里

需要设置用户名和邮箱(用户标识，必要)

当安装Git后，首先要做的事情就是设置用户名称和email地址。这是非常重要的，因为git提交都会使用该信息，他被永远嵌入到了你的提交中。

创建本地仓库的方法有两种：一种是创建全新的仓库，另一种是克隆远程的仓库

  1. 创建全新的仓库，需要git管理项目的根目录执行

  2. 克隆远程仓库

    git文件操作

    文件四种状态

    版本控制就是对文件的版本进行控制，要对文件进行修改，提交等操作，首先要知道文件当前在什么状态下，不然可能会提交了现在还不想提交的文件，或者想要提交的文件没有提交上

    • Untracked : 未跟踪，此文件在文件夹中，但没有加入到git库，不参与版本控制，通过 git add状态变为Staged
    • Unmodify: 文件已经入库，未修改，即版本库中的文件快照内容与文件夹完全一致，这种类型的文件有两种去处，如果他被修改，变为Modified, 如果使用git rm移除版本库，则成为Untracked
    • Modified: 文件已修改，仅仅是修改，并没有进行其他的操作，这个文件也有两个去处，通过git add可进入暂存状态staged状态，使用gitcheckout，则丢弃修改过，返回到unmodify状态，这个git checkout即为从库里取出文件，并覆盖当前修改。
    • Staged: 暂存状态，执行git commit,则将修改同步到库中，这时，库中的文件又和本地文件变为一致，文件为unmodify状态，执行git reset HEAD filename取消暂存，文件状态为Modified

忽略文件

有些时候我们不想把某些文件纳入版本控制中，比如数据库文件，临时文件，设计文件等，在主目录下，建立 “.gitignore” 文件，此文加你有如下规则:

  1. 忽略文件中的空行，或者以”#”(井号)开始的行会被忽略
  2. 可以使用linux通配符，例如：星号(*)代表多个任意字符，问号(?)代表一个字符，方括号([abc])代表可选字符范围，大括号({String 1,String 2。。。}) 代表可选的字符串等。
  3. 如果名称的最前面有一个感叹号(!),表示例外规则，将不会被忽略。
  4. 如果名称的最前面是由一个路径分隔符(/)，表示要忽略的文件在此目录下，而子目录中的文件不被忽略。
  5. 如果名称的最后面是一个路径的分隔符(/),表示要忽略的是此目录下，该名称的子目录，而非邮件(默认文件或者目录都忽略)

配置ssh

github 是有墙的，比较慢，在国外的话，我们一般使用gitee，公司有时候会搭建自己的gitlab服务器。

在 C:/用户/Administrative/.ssh 文件夹里打开终端

然后给自己的码云或者github绑定，免密登录。

之后可以使用 git 绑定IDEA

git分支

分支最主要的是master(可以自行类比二叉树)

git 分支常用指令

多个分支如果并行执行，就会导致我们的代码不冲突，也就是同时存在多个多个版本。

web-api —-A开发

web -admin —–B开发 (会调用A,会修改A的内容)

web -app —-C开发 会调用B和A的代码

如果同一个文件在合并分支时都被修改了则会引起冲突，解决的办法是，我们可以修改冲突文件后重新提交，选择要保留他的代码还是你的代码。

master主分支理应是很稳定的，用来发布新的版本，一般不允许在上面工作，工作一般情况下在新建的dev分支上工作，工作完后，如果要发布，可以合并到主分支上。
